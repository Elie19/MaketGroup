# Audit technique & sécurité — MaketGroup

Date: 2026-03-27
Périmètre: application front React/Vite + server Express + intégration Supabase

## Résumé exécutif

Statut global: **à corriger avant mise en production**.

Constats principaux:
1. **Le projet ne compile pas** (imports `isConfigured` inexistants).
2. **Journalisation sensible côté client** (sessions/auth events loggés en clair).
3. **Configuration Supabase de secours fragile** (valeurs placeholder codées en dur, risque de faux positifs/negatifs).
4. **Robustesse perfectible sur certaines opérations data/realtime** (absence de validation et de garde sur des mappings).

## Détail des constats

### 1) Build cassé: export manquant `isConfigured` (Critique)
- `App.tsx` et `Home.tsx` importent `isConfigured` depuis `src/supabase.ts`.
- `src/supabase.ts` n’exporte pas `isConfigured`.
- Impact: blocage `tsc --noEmit` et risque de pipeline CI rouge.

**Preuve:**
- Import dans `src/App.tsx`.
- Import dans `src/pages/Home.tsx`.
- Fichier source `src/supabase.ts` sans export.

**Action recommandée:**
- Ajouter un export explicite `isConfigured` dans `src/supabase.ts` (basé sur URL + clé valides), ou retirer l’usage côté UI.

---

### 2) Logs d’auth en clair dans le store (Élevée)
- `useAuthStore.init()` trace `session` et les événements auth via `console.log`.
- En pratique, les objets de session peuvent contenir des métadonnées sensibles (selon SDK/env).
- Impact: fuite d’informations dans la console utilisateur / collecte de logs front.

**Preuve:**
- `console.log('Initial session:', session)`.
- `console.log('Auth state change event:', event, session)`.

**Action recommandée:**
- Supprimer ces logs, ou les protéger derrière un flag strictement dev + redaction.

---

### 3) Stratégie de fallback Supabase insuffisamment sûre (Moyenne)
- Le client prend `import.meta.env` sinon `supabase-config.json`.
- Le fichier JSON contient des placeholders (`supabaseAnonKey: "002001"`).
- Un warning existe, mais le client est créé même en cas de config invalide.
- Impact: comportements non déterministes, erreurs tardives, expérience dégradée.

**Preuve:**
- `src/supabase.ts`.
- `supabase-config.json`.

**Action recommandée:**
- Centraliser une validation stricte (`isConfigured`) et court-circuiter proprement les services en mode non configuré.
- Exclure toute valeur placeholder des flux “normaux”.

---

### 4) Défauts de robustesse data (Moyenne)
- `transactionService.getSellerTransactions` fait `return data.map(...)` sans garde si `data` est null/undefined.
- `chatService.sendMessage` déduit `participants` via `chatId.split('_')` sans validation de format.
- Impact: erreurs runtime si payload inattendu; surface d’incohérence métier.

**Preuve:**
- `src/services/transactionService.ts`.
- `src/services/chatService.ts`.

**Action recommandée:**
- Ajouter des gardes (`if (!data) return []`).
- Normaliser/valider le format de `chatId` avant persistance.

## Vérifications exécutées

- `npm run lint` → **échec** (erreurs TypeScript bloquantes).
- `npm audit --audit-level=moderate` → **non concluant** (endpoint npm audit inaccessible depuis cet environnement).

## Priorisation proposée

1. **P0 (immédiat):** corriger `isConfigured` pour rétablir la compilation.
2. **P1:** retirer les logs d’auth sensibles.
3. **P1:** durcir la config Supabase (validation + mode dégradé explicite).
4. **P2:** ajouter gardes runtime et validation des IDs métier.

## Conclusion

Le socle est proche d’un MVP fonctionnel, mais l’état actuel montre un **risque de non-fiabilité en déploiement** (build cassé) et des **points sécurité/observabilité à assainir**.
